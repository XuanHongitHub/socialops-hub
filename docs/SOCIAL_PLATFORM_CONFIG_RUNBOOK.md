# Social Platform Config Runbook

This file is the source of truth for configuring social platforms in SocialOps Hub. Update it whenever a platform setup fails, a UI flow changes, or a new workaround is found.

## Golden Flow

1. Confirm account policy:
   - Login email can differ by platform.
   - Public contact stays brand email (`admin@bugsell.com` / `support@bugsell.com`).
   - Do not force private login email into public profile details.
2. Create or reuse one developer app per platform first.
3. Add exact local callback URI generated by the app, not a guessed URI.
4. Put keys into `project/aitoearn-backend/apps/aitoearn-server/config/config.yaml` using the schema used by the running image.
5. Disable relay unless `RELAY_API_KEY` is valid.
6. Restart server/web/nginx after config changes.
7. Test auth URL by API before clicking in UI.
8. Complete OAuth in the browser with the target account.
9. Verify account appears in `http://localhost:18080/en/accounts`.

## Browser Ops Rules

- Use a clean browser/profile for platform setup when possible.
- Before clicking, confirm visible account, URL, and active tab.
- Prefer direct URLs from API output over hunting UI buttons.
- If a click does nothing, inspect network/logs before retrying repeatedly.
- If a page is unfocused or stale, reload once, then reopen the exact URL.
- For OAuth, never start multiple sessions in parallel; old `state` values expire fast.
- If verification is required, stop and let the human enter the code.
- If popup blocked, open the auth URL directly in the active browser.

## Local App Checks

```powershell
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
docker logs --tail 120 aitoearn-server
Invoke-RestMethod http://localhost:18080/api/health
```

Get the local auto-login token:

```powershell
$tok = (docker exec aitoearn-web sh -lc 'cat /data/init/token.txt').Trim()
```

Test YouTube auth without UI:

```powershell
Invoke-RestMethod -Headers @{Authorization="Bearer $tok"} `
  -Uri http://localhost:18080/api/v2/channels/accounts/auth/youtube `
  -Method Get
```

Expected success: `code: 0` plus `data.url` pointing to `accounts.google.com`.

## Config Rules

- Do not edit root install/build state for config-only work.
- Do not put secrets into docs; use placeholders here, real values only in local config/env.
- Match config keys to the running image, not just repo source.
- If source and Docker image disagree, inspect files inside container:

```powershell
docker exec aitoearn-server sh -lc 'find /app/apps -name "*youtube*config.js" -o -name "*platforms*config.js"'
docker exec aitoearn-server sh -lc 'cat /app/apps/aitoearn-server/src/core/channels/platforms/youtube/youtube.config.js'
```

- Restart after YAML changes:

```powershell
docker compose -f docker-compose.yml -f docker-compose.local.yml restart aitoearn-server aitoearn-web nginx
```

## Relay Rule

Relay is useful only with a valid platform relay API key. If `relay:` exists but `apiKey` is empty, local OAuth may be bypassed and calls can return `401 Unauthorized` from `aitoearn.ai`.

Fix for local/self-owned OAuth:

```yaml
# Remove/disable relay block when no RELAY_API_KEY exists.
# relay:
#   serverUrl: https://aitoearn.ai/api
#   apiKey: ""
#   callbackUrl: http://localhost:8080/api/v2/channels/relay/callback
```

## Troubleshooting Log

### YouTube auth returned `code:401 Unauthorized`

Cause:

- Local config had `relay:` enabled with empty API key.
- Server forwarded auth to `https://aitoearn.ai/api/...` instead of local OAuth.

Fix:

- Remove/disable `relay` block for local OAuth.
- Use v2.5 config keys: `clientId`, `clientSecret`, `redirectUri`.
- Restart server/web/nginx.
- Verify auth endpoint returns Google OAuth URL.

### Source repo config did not match Docker image

Cause:

- Local source used one schema; pulled Docker image used newer `channels` implementation/schema.

Fix:

- Inspect `/app/apps/.../*.config.js` inside the running container.
- Patch local mounted `config.yaml` to match runtime schema.

### UI showed auth failure but API token looked present

Cause:

- Browser had stale auth/session/cookie data, while app token source changed after image/config restart.

Fix:

- Use `/data/init/token.txt` from `aitoearn-web` for direct API testing.
- Reload/clear app page only after API endpoint succeeds.


### YouTube Google error `redirect_uri_mismatch`

Cause:
- Google OAuth client still had old AiToEarn callback: `/api/plat/youtube/auth/callback`.
- Current Docker image generated v2 callback: `/api/v2/channels/accounts/auth/youtube/callback`.

Fix:
- Add exact generated redirect URI to Google Cloud OAuth Web client.
- Save client, then start a fresh auth session; old sessions remain invalid.

Verification:
- OAuth page reaches consent screen instead of error 400.
- Connected account saved in `aitoearn.account` as `youtube_108887968779802746573_UCFSj7rfdvBRMUXrv7Su__fg` / `BugSell`.


### Platform auth URL contains empty `client_id`

Cause:
- SocialOps Hub can generate an OAuth URL even when a platform has no developer app keys.
- Empty `client_id` URLs are not connectable and should not be opened except for diagnosis.

Fix:
- Check `docker-compose.local.yml` and `config.yaml` first.
- Only start OAuth when the platform has non-empty client ID and secret/app secret.
- For Pinterest, TikTok, X, LinkedIn, Facebook, Instagram, Threads: create the official developer app first, then add the exact redirect URI generated by the app.

Verification:
- Auth URL must include a non-empty `client_id` / `client_key`.
- OAuth consent must reach the provider app authorization page, not an invalid-client page.
